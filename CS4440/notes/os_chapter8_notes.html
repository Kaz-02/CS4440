01 What is a Deadlock?
Definition

A deadlock is a liveness failure where every thread in a set is waiting for an event that can only be caused by another waiting thread in the same set — making progress impossible.

In a multiprogramming environment, threads compete for a finite number of resources (CPU cycles, files, I/O devices, mutex locks, semaphores). The normal lifecycle of resource usage is:

01
Request

Thread requests a resource. If unavailable, it enters a waiting state.

02
Use

Thread operates on the resource after it has been granted.

03
Release

Thread releases the resource, making it available to others.

⚠

Key Insight: Deadlocks can be insidious — they may only occur under specific scheduling circumstances, making them difficult to identify and test for.

Classic example: Thread 1 holds first_mutex and waits for second_mutex, while Thread 2 holds second_mutex and waits for first_mutex.

deadlock_example.c 
/* Thread One */
void do_work_one() {
  pthread_mutex_lock(&first_mutex);   // acquires first
  pthread_mutex_lock(&second_mutex);  // waits for second ← blocked here
  /** Do some work **/
  pthread_mutex_unlock(&second_mutex);
  pthread_mutex_unlock(&first_mutex);
}

/* Thread Two */
void do_work_two() {
  pthread_mutex_lock(&second_mutex);  // acquires second
  pthread_mutex_lock(&first_mutex);   // waits for first ← blocked here
  /** Do some work **/
  pthread_mutex_unlock(&first_mutex);
  pthread_mutex_unlock(&second_mutex);
}
02 Deadlock vs. Livelock
Aspect Deadlock Livelock
Thread State Blocked and waiting Active, continuously retrying (not blocked)
Progress None — fully stuck None — busy but going nowhere
CPU Usage Zero or very low Very high
Detection Easy (system hangs) Difficult (looks like the system is busy)
Recovery Hard; often requires external termination Hard; requires randomness or breaking symmetry
Analogy Jammed gears — no movement at all Two people in a hallway stepping side-to-side forever
Livelock Solution

Introduce randomness (like Ethernet backoff). Have each thread retry after a random delay to break the symmetric failure pattern. This prevents both threads from always making the same conflicting move simultaneously.

03 Four Necessary Conditions

A deadlock can only arise if all four of these conditions hold simultaneously. Eliminating any one prevents deadlock.

Condition 1
Mutual Exclusion

At least one resource is non-sharable. Only one thread can use it at a time.

Condition 2
Hold and Wait

A thread holds at least one resource while waiting to acquire additional resources held by other threads.

Condition 3
No Preemption

A resource can only be released voluntarily by the thread holding it. It cannot be forcibly taken away.

Condition 4
Circular Wait

A set {T₀, T₁, …, Tₙ} exists such that T₀ waits for T₁'s resource, T₁ waits for T₂'s, …, Tₙ waits for T₀'s.

04 Resource-Allocation Graph (RAG)

A directed graph that visualizes the state of resource assignments and thread requests in the system.

Vertices
T Threads — drawn as circles
R Resources — drawn as rectangles (dots = instances)
Edges
Request edge Tᵢ → Rⱼ : thread waiting for a resource
Assignment edge Rⱼ → Tᵢ : resource allocated to thread
Claim Edge (Avoidance)
Dashed line Tᵢ --→ Rⱼ: thread may request this resource in the future (used in RAG avoidance algorithm).
Conversion
When a request is granted, a request edge instantly becomes an assignment edge.
Cycle Detection Rules
✅ No cycle → system is definitely NOT deadlocked.
⚠️ Cycle present, multiple instances → system MAY be deadlocked. (Another thread outside the cycle might release a resource.)
🔴 Cycle present, single-instance resources only → deadlock is CONFIRMED. Every thread in the cycle is deadlocked.
05 Methods for Handling Deadlocks
Method 01
Prevention
Preventative Medicine

Constrain resource requests so at least one of the four conditions can never hold. Makes deadlock structurally impossible.

Method 02
Avoidance
Proactive Screening

Use a priori knowledge of each thread's maximum resource needs to dynamically decide if a request is safe to grant.

Method 03
Detection & Recovery
Emergency Intervention

Allow deadlocks to occur, then run an algorithm to detect and recover from them.

💡

Real-world note: Most OS (Linux, Windows) ignore the problem — the overhead of prevention/avoidance algorithms is considered more costly than handling the occasional deadlock manually. Responsibility falls on application developers. Exception: Database systems implement full detection & recovery because data integrity is paramount.

06 Deadlock Prevention

Attack one of the four necessary conditions to make deadlock impossible.

✗ Mutual Exclusion

Largely impossible. Some resources (like mutex locks) are intrinsically non-sharable by nature.

✗ Hold and Wait

Require threads to request all resources at once, or release all before requesting new ones. Drawbacks: low resource utilization and starvation risk.

✗ No Preemption

Forcibly take resources from a waiting thread. Only feasible for resources with easily saved/restored state (e.g., CPU registers). Not practical for locks.

✓ Circular Wait — Most Practical

Impose a total ordering on all resource types. Require all threads to request resources in increasing order of enumeration. Example: if F(first_mutex)=1 and F(second_mutex)=5, every thread must acquire first_mutex before second_mutex.

⚠

Important caveat: Even with lock ordering defined, if the ordering is not enforced in code (e.g., locks acquired based on runtime values), a deadlock can still occur. Linux's lockdep tool dynamically monitors acquisition patterns and reports inconsistencies.

07 Deadlock Avoidance & Safe State
Safe State

A state is safe if the system can allocate resources to each thread (up to its maximum) in some order and still avoid deadlock. This ordering is called a safe sequence.

Key principle: Safe ⊃ Unsafe ⊃ Deadlock. An unsafe state is not a deadlock, but it may lead to one. The goal is to never move from a safe to an unsafe state.

Avoidance requires a priori information: each thread must declare the maximum number of each resource type it may need. The system then dynamically evaluates every request using one of two algorithms:

RAG Algorithm

For single-instance resources. Uses claim edges (dashed). Grant a request only if doing so creates no cycle. O(n²) cycle detection.

Banker's Algorithm

For multi-instance resources. Simulates granting the request, then runs the Safety Algorithm to check if a safe sequence still exists.

08 Banker's Algorithm

Named after how a banker ensures they never lend out funds in a way that prevents satisfying all future customer needs.

Data structures (n = threads, m = resource types):

Available [m]

Number of available instances of each resource type.

Max [n×m]

Maximum demand of each thread for each resource type.

Allocation [n×m]

Currently allocated resources for each thread.

Need [n×m]

Remaining need. Need = Max − Allocation

Request algorithm steps:

1️⃣Pre-check: Requestᵢ ≤ Needᵢ AND Requestᵢ ≤ Available. Otherwise error/wait.
2️⃣Hypothetical allocation: Pretend to grant the request; update Available, Allocation, Need.
3️⃣Safety check: Run the Safety Algorithm on the new hypothetical state.
✅If SAFE → make allocation permanent. If UNSAFE → discard hypothetical state; thread must wait.

Example (5 threads, 3 resource types A:10, B:5, C:7):

Thread Allocation Max Need (Max−Alloc)
ABC ABC ABC
T₀010753743
T₁200322122
T₂302902600
T₃211222011
T₄002433431

Available = (3, 3, 2). Safe sequence: <T₁, T₃, T₄, T₂, T₀> Safe

If T₁ requests (1, 0, 2): New available = (2, 3, 0), new safe sequence <T₁, T₃, T₄, T₀, T₂> — request granted.

If T₀ then requests (0, 2, 0): Available becomes (2, 1, 0) — no safe sequence possible — Unsafe request denied.

09 Detection & Recovery
Single-Instance Resources

Use a Wait-For Graph (resource nodes removed; edge Tᵢ → Tⱼ if Tᵢ waits for a resource held by Tⱼ). Deadlock iff a cycle exists.

Multi-Instance Resources

Use a variant of the Banker's Safety Algorithm. If threads remain that cannot finish, they are deadlocked.

When to Run?

Option A: every denied request (high responsiveness, high overhead). Option B: fixed intervals or when CPU drops below ~40% (low overhead, slower response).

Recovery options:

Option 1: Process Termination
  • Brute force: Abort all deadlocked processes (effective but costly — partial work lost)
  • Surgical: Abort one process at a time until cycle breaks (requires re-detecting after each kill)
Option 2: Resource Preemption
  • Action: Forcibly take a resource from one process and give it to another
  • Victim selection: Consider priority, runtime, resources held
  • Rollback: Victim must be rolled back to a safe state before re-execution
  • Starvation risk: Track rollback count to avoid always picking the same victim
