Problem Concurrency

Background: Concurrent & Parallel Execution

  • Concurrent execution: CPU scheduler がプロセスを高速に切り替え。1つのプロセスが途中で中断される可能性がある。
  • Parallel execution: 複数の命令が別々の processing core で同時に実行される。
  • どちらの場合も、共有データの 整合性 (integrity) が脅かされる。

Producer-Consumer Problem（有界バッファ問題）

Producer と Consumer が固定サイズの buffer を共有する古典的な問題。count 変数で埋まっているスロット数を管理する。

⚠ 問題点
Producer の count++ と Consumer の count-- は、それぞれ単体では正しく動くが、同時に実行すると結果が壊れる。

なぜ count++ は危険か？（機械語レベル）

count++ は実は3命令に展開される:

// count++ の実際の動作
register1 = count          // (1) メモリから読む
register1 = register1 + 1  // (2) インクリメント
count = register1          // (3) メモリに書く

Race Condition の例（count = 5 の場合）

時刻実行内容値
T0Producer: register1 = countregister1 = 5
T1Producer: register1 = register1 + 1register1 = 6
T2Consumer: register2 = countregister2 = 5
T3Consumer: register2 = register2 - 1register2 = 4
T4Producer: count = register1count = 6 ✗
T5Consumer: count = register2count = 4 ✗ （正解は5）
🔑 Race Condition とは
複数のプロセスが共有データに同時アクセスし、実行順序によって結果が変わる状態。結果が unpredictable（予測不能）になる。
Core Concept

Critical Section とは

共有変数・テーブル・ファイルなどの共有リソースを操作するコードの部分。あるプロセスがここを実行している間は、他のプロセスは入れない。

有効な解が満たすべき3条件

① Mutual Exclusion（相互排除）

プロセス Pᵢ が critical section にいる間、他のどのプロセスもそこに入れない。

② Progress（進行）

Critical section が空いているとき、次に入るプロセスの選択を無限に延期してはならない。

③ Bounded Waiting（有界待機）

あるプロセスが要求を出してから許可されるまでに、他のプロセスが割り込める回数に上限がある。

Software Classic

アイデア

2プロセス（Pᵢ と Pⱼ）向けの古典的アルゴリズム。2つの共有変数を使う:

  • int turn — 誰のターンか（どちらが critical section に入れるか）
  • boolean flag[2] — flag[i] = true は「Pᵢ が入る準備ができた」を意味
Process Pᵢ
// Entry Section
flag[i] = true;
turn = j;
while (flag[j] && turn == j)
  ; // wait

/* CRITICAL SECTION */

// Exit Section
flag[i] = false;
Process Pⱼ
// Entry Section
flag[j] = true;
turn = i;
while (flag[i] && turn == i)
  ; // wait

/* CRITICAL SECTION */

// Exit Section
flag[j] = false;
💡 鍵となるロジック
「相手に譲る」設計: Pᵢ は turn = j とセットすることで「もしどちらも入りたいなら相手を優先する」と宣言する。これが Mutual Exclusion の鍵。

理論的には完璧だが…

⚠ 現代の問題点
現代のプロセッサやコンパイラは命令を reorder（並べ替え）する。データ依存のない命令は最適化のため順番が変わる可能性がある。

例: flag[i] = true; turn = j; が turn = j; flag[i] = true; に並べ替えられると、両プロセスが同時に critical section に入れてしまう。

→ 現代のアーキテクチャでは動作が保証されない。
Hardware Atomic
🔑 Atomic とは
中断・分割できない一つの単位として実行される操作。ハードウェアバスレベルで保証される。現代のシステムでは「exclusive cache content」方式を使い、他のプロセッサのキャッシュを無効化してから操作を行う。

① Memory Barriers（メモリバリア）

命令の reorder を防ぐ「壁」。バリアより前の全ての load/store が完了してから、後の命令が始まる。

x = 100;
memory_barrier();  // この壁を挟むことで順序が保証される
flag = true;

② test_and_set()

boolean 変数を trueにセットし、元の値を返す。この読み取りとセットが atomic（不可分）に行われる。

// ハードウェアが提供するアトミック関数
bool test_and_set(bool *lock) {
  bool old = *lock;
  *lock = true;   // 常にtrueにセット
  return old;     // 元の値を返す
}

// 使い方（Mutex実装）
do {
  while (test_and_set(&lock))
    ; // old=trueなら他がロック中 → spin
  /* Critical Section */
  lock = false;  // ロック解放
} while (true);
💡 動作の仕組み
最初に lock=false のとき: test_and_set が false（元の値）を返しながら lock を true に変える → ループを抜けて CS へ入る。lock=true なら true を返す → ループし続けて待つ（spin）。

③ compare_and_swap()（CAS）

メモリの値が 期待値と一致すれば新しい値に書き換える。不一致なら何もしない。元の値を返す。

int compare_and_swap(int *lock, int expected, int new_value) {
  int original = *lock;
  if (*lock == expected)
    *lock = new_value;
  return original;  // 元の値を返す
}

// 使い方（lock=0がアンロック、1がロック）
while (true) {
  while (compare_and_swap(&lock, 0, 1) != 0)
    ; // 0→1に交換できなければ待つ（既にロック中）
  /* Critical Section */
  lock = 0;  // ロック解放
}
📌 Key Insight
これら atomic instructions が ほぼすべての高レベル同期ツールの構成要素（building blocks）となる。Mutex も Semaphore も、内部では CAS や test_and_set を使っている。
Software Tool I

概念

Mutex（Mutual Exclusion）lock は、hardware instructions 上に構築されたシンプルなソフトウェアツール。Critical section を保護し race condition を防ぐ。

プロトコル

  • acquire(): Critical section に入る前にロックを取得する
  • release(): Critical section を出たらロックを解放する
// シンプルな実装（boolean available）
acquire() {
  while (!available)
    ; // busy wait（ここが問題）
  available = false;
}

release() {
  available = true;
}

Spinlock と Busy Waiting の問題

⚠ Spinlock の欠点
プロセスがロック待ちの間、ループを回し続けて CPU サイクルを消費し続ける（Busy Waiting）。これを Spinlock と呼ぶ。

でも Spinlock が有効な場面もある

「ブロック（スリープ）」はコンテキストスイッチ2回分のオーバーヘッドがある（スリープ時と復帰時）。

💡 ガイドライン
ロックが保持される時間 < コンテキストスイッチ2回の時間 ならSpinlockの方が効率的。マルチコアシステムでは、あるコアがスピンしながら待つ間、別のコアは CS の仕事を進められる。シングルコアには不適。
Software Tool II Dijkstra

定義

Semaphore S は整数変数で、以下の2つの atomic 操作を通じてのみアクセスされる（Dijkstra が考案）:

  • wait(S)（元はオランダ語で "proberen"=「試みる」の P）: S をデクリメント。S ≤ 0 になるとプロセスはブロックされる。
  • signal(S)（元は "verhogen"=「増やす」の V）: S をインクリメント。待機中のプロセスがあれば1つ起こす。
wait(S) の定義
wait(S) {
  while (S <= 0)
    ; // busy wait
  S--;
}
signal(S) の定義
signal(S) {
  S++;
}

2種類の Semaphore

  • Binary Semaphore: 値が 0 か 1 のみ。Mutex lock と同等に機能する。
  • Counting Semaphore: 値が任意の範囲。有限リソースへのアクセス管理に使う。

応用例① — リソースプール管理（Counting Semaphore）

ライセンスが3枚ある場合、S=3 で初期化。wait()で取得、signal()で返却。S=0 になったら次のプロセスは自動でブロック。

応用例② — 実行順序の制御

P1 の S1 を P2 の S2 より先に実行したい場合:

Process P1
S1;
signal(synch);
Process P2
wait(synch);
S2;

synch を 0 で初期化 → P2 は P1 が signal するまで S2 を実行できない。

Semaphore の危険性

✗ 順序の誤り

signal(mutex);
// critical section
wait(mutex);

Mutual exclusion が破れる。複数プロセスが CS に入れてしまう。

✗ signal を wait で置換

wait(mutex);
// critical section
wait(mutex);

2回目の wait で永久にブロック（Deadlock）。

✗ wait/signal を省略

// wait省略
// critical section
signal(mutex);

Mutual exclusion が破れるか、永久にブロックされる。

Software Tool III ADT

なぜ Monitor が必要か

Semaphore は強力だが、プログラマが wait()/signal() を正しく呼ぶ規律に完全に依存する。ちょっとしたミスが Deadlock や mutual exclusion 違反を起こす。より安全で使いやすい仕組みが必要。

Monitor とは

Monitor は高レベルの同期構造体（Abstract Data Type）。以下を一つにカプセル化する:

  • 共有変数（shared variables）
  • それらを操作するプロシージャ（procedures）
  • 組み込みの Mutual Exclusion（自動排他制御）

主要な特徴

  • 自動 Mutual Exclusion: 一度に1つのプロセスだけが monitor 内で active になれる。コンパイラが強制するので、プログラマが wait()/signal() を呼ぶ必要がない。
  • Condition Variables: より複雑な同期のために、プロセスは特定の条件を待てる。

Condition Variables の操作

  • x.wait(): 別のプロセスが x を signal するまで呼び出したプロセスをサスペンドする。同時に monitor ロックをアトミックに解放し、他のプロセスが入れるようにする。
  • x.signal(): 条件 x を待っているプロセスを1つだけ再開させる（待っていなければ何もしない）。
💡 Semaphore との違い
Semaphore の signal() は待機プロセスがいなくてもカウントが増える（後で使える）。Monitor の x.signal() は待機プロセスがいなければ効果なし（記憶しない）。
Danger Liveness

Liveness とは

Liveness = プロセスが実行中に前進（progress）できることを保証するためにシステムが満たすべき性質の集合。Liveness failure = プロセスが無限に前進できない状態。

① Deadlock（デッドロック）

2つ以上のプロセスが、互いに相手が引き起こすイベントを永遠に待ち続ける状態。

// 例: P0とP1がS,Qという2つのsemaphoreを保持し合う
P0: wait(S) → wait(Q) → ... → signal(S) → signal(Q)
P1: wait(Q) → wait(S) → ... → signal(Q) → signal(S)
// P0がSを保持してQを待ち、P1がQを保持してSを待つ → 永久に膠着

② Priority Inversion（優先度逆転）

高優先度プロセスが低優先度プロセスの保持するロックを必要とし、さらにその低優先度プロセスが中優先度プロセスに preempt されて動けない状態。

  • H（高優先度）がロックを待つ
  • L（低優先度）がロックを保持しているが M（中優先度）に preempt される
  • 結果: H が M より低い実効優先度で動くことになる

🚀 実例: 1997年 Mars Pathfinder の Sojourner 探査機

火星表面で頻繁にシステムリセットが発生。原因は Priority Inversion だった。

bc_dist（高優先度・通信） → ASI/MET（低優先度・気象）が保持する mutex を待つ → 複数の中優先度タスク（Communications）に ASI/MET が preempt される → watchdog timer がデッドロックを検知してシステムをリセット。

解決策: VxWorks RTOS の Priority Inheritance Protocol を有効にするフラグを、地球から探査機にコマンド送信。L が一時的に H の優先度を引き継ぐことで M の preemption を防ぐ。

🔑 章のまとめ
Synchronization は「parallelism の性能を犠牲にせずに data integrity を維持する技術」。低レベルの atomic hardware 命令から高レベルの Monitor まで、抽象化の層があり、それぞれトレードオフが存在する。有効な解は常に Mutual Exclusion・Progress・Bounded Waiting の3条件を満たさなければならない。
