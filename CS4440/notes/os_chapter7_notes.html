01 The Core Challenge
The Problem

When multiple concurrent processes share data, race conditions arise — unpredictable outcomes from unsynchronized access. The core task is to enforce mutual exclusion and orderly cooperation.

Three "classic problems" serve as standard benchmarks for any new synchronization scheme. They expose the fundamental challenges of concurrency: mutual exclusion, deadlock, and starvation.

Classic Problem 1

Bounded-Buffer — coordinating a producer and consumer sharing a fixed-size buffer.

Classic Problem 2

Readers-Writers — managing concurrent reads and exclusive writes on shared data.

Classic Problem 3

Dining Philosophers — allocating limited resources among competing processes without deadlock or starvation.

02 Bounded-Buffer Problem

A fixed-size buffer (n slots) is shared between a producer (generates items) and a consumer (uses items). The challenge: the producer must not add to a full buffer, and the consumer must not take from an empty one.

Semaphore Variables
mutex = 1Provides mutual exclusion for accesses to the buffer pool.
empty = nCounts the number of empty buffer slots. Producer waits when this reaches 0.
full = 0Counts the number of full buffer slots. Consumer waits when this reaches 0.
producer 
// produce item...
wait(empty);  // wait if empty=0
wait(mutex);  // wait if mutex=0

// add item to buffer

signal(mutex); // unlock buffer
signal(full);  // signal full slot
consumer 
wait(full);   // wait if full=0
wait(mutex);  // wait if mutex=0

// remove item from buffer

signal(mutex); // unlock buffer
signal(empty); // signal empty slot
// consume item...
⚠

Order matters: Both producer and consumer acquire mutex after the counting semaphore. Reversing the order (locking mutex first) can cause deadlock if the buffer is full/empty.

03 Readers-Writers Problem

A shared database is accessed by readers (read-only) and writers (update). Rules: multiple readers may access simultaneously, but a writer requires exclusive access — no other reader or writer may be active.

Shared Variables
rw_mutex = 1Provides mutual exclusion for writers; also used by the first/last reader.
mutex = 1Protects the read_count variable from race conditions.
read_count = 0Tracks how many readers are currently active.
reader 
wait(mutex);
read_count++;
if (read_count == 1)  // first in
  wait(rw_mutex);  // lock writers
signal(mutex);

/* ... reading ... */

wait(mutex);
read_count--;
if (read_count == 0)  // last out
  signal(rw_mutex); // allow writer
signal(mutex);
writer 
wait(rw_mutex);

/* ... writing ... */

signal(rw_mutex);
Key Logic (Reader Priority)
🔵The first reader in acquires rw_mutex, locking out all writers for the duration.
🔵Subsequent readers enter freely since they only check read_count under mutex.
🔴The last reader out releases rw_mutex, finally allowing a writer to proceed.
⚠️Starvation risk: If readers arrive continuously, writers may never get access (writer starvation).
04 Dining Philosophers Problem

Five philosophers sit at a circular table, alternating between thinking and eating. One chopstick lies between each pair. To eat, a philosopher must pick up both adjacent chopsticks. The goal: design a protocol with no deadlock and no starvation.

OS Mapping:

Parable OS Concept Meaning
PhilosophersProcesses / ThreadsActive entities requiring resources to proceed.
ChopsticksShared ResourcesLimited items requiring mutual exclusion (e.g., memory, locks).
Picking up chopstickResource AcquisitionRequesting and locking a resource (semaphore/mutex).
EatingCritical SectionProcess has exclusive access and is performing work.
ThinkingProcess ExecutionRunning but not needing shared resources.
naive_solution.c — FLAWED ⚠ 
semaphore chopstick[5]; // all init to 1

/* philosopher i */
wait(chopstick[i]);           // pick up LEFT
wait(chopstick[(i+1) % 5]);  // pick up RIGHT
/* ... eat ... */
signal(chopstick[i]);
signal(chopstick[(i+1) % 5]);

// ⚠ Deadlock: if ALL five pick up their left chopstick
// simultaneously, none can ever pick up the right one.

Solutions:

1. Limit Concurrency

Allow at most 4 philosophers at the table at once. Guarantees at least one can always eat.

2. Atomic Pickup

Require a philosopher to pick up both chopsticks only if both are available simultaneously (atomic operation).

3. Asymmetric Rule

Odd philosophers pick left then right; even philosophers pick right then left. Breaks circular wait.

4. Resource Hierarchy

Number the chopsticks and require philosophers to always pick up the lower-numbered one first. Prevents circular wait.

Monitor Solution (best approach):

monitor DiningPhilosophers 
enum {THINKING, HUNGRY, EATING} state[5];
condition self[5];

void pickup(int i) {
  state[i] = HUNGRY;
  test(i);
  if (state[i] != EATING) self[i].wait(); // block
}

void putdown(int i) {
  state[i] = THINKING;
  test((i + 4) % 5); // check left neighbor
  test((i + 1) % 5); // check right neighbor
}

void test(int i) {
  // can eat only if HUNGRY and both neighbors NOT EATING
  if (state[(i+4)%5] != EATING &&
      state[i] == HUNGRY &&
      state[(i+1)%5] != EATING) {
    state[i] = EATING;
    self[i].signal();
  }
}
💡

Monitor advantage: Centralizes the complex logic of acquiring and releasing resources, making it far less error-prone than managing individual semaphores. Caveat: While deadlock-free, this solution does not guarantee fairness — a philosopher could still starve.

05 Kernel Synchronization: Windows vs. Linux

The classic problem principles are directly applied within OS kernels. These are low-level mechanisms that protect the kernel itself — the foundation upon which user-level APIs are built.

Windows
Core Mechanism

Uses Dispatcher Objects for thread synchronization (mutexes, semaphores, events). Objects are in a signaled (available, thread won't block) or nonsignaled (unavailable, thread will block) state.

Short Critical Sections

Uses spinlocks on multiprocessor systems. The kernel ensures a thread holding a spinlock is never preempted.

User-Mode Optimization

Critical-section objects are efficient user-mode mutexes that only allocate a kernel mutex when there is actual contention.

Linux
Kernel Model

Fully preemptive kernel since version 2.6. Any task can be preempted even while running in kernel mode.

Short Critical Sections

Primary locking mechanism on SMP systems is the spinlock, held for very short durations. For longer durations, mutex locks are used (sleeping task).

Atomic Integers

Provides atomic_t for simple lock-free counter operations: atomic_set, atomic_add, atomic_inc, atomic_read.

Preemption Control

preempt_disable() / preempt_enable() — a preempt_count per-task tracks lock acquisitions to prevent preemption while a lock is held.

06 POSIX Synchronization

The POSIX API provides a standard set of user-level synchronization tools for programmers on UNIX, Linux, and macOS.

Mutex Locks (pthread_mutex_t)

The fundamental technique for protecting critical sections.

  • pthread_mutex_init()
  • pthread_mutex_lock()
  • pthread_mutex_unlock()
Semaphores (sem_t)

Named (sem_open()): shared between unrelated processes.
Unnamed (sem_init()): shared between threads of the same process.

  • sem_wait() — decrement (P)
  • sem_post() — increment (V)
Condition Variables (pthread_cond_t)

Used with a mutex to wait for a specific condition to become true. Avoids busy-waiting.

  • pthread_cond_wait()
  • pthread_cond_signal()
POSIX named semaphore 
// shared between unrelated processes
sem_t *sem;
sem = sem_open("SEM", O_CREAT, 0666, 1);

sem_wait(sem);   // acquire
/* critical section */
sem_post(sem);   // release
POSIX unnamed semaphore 
// shared between threads of same process
sem_t sem;
sem_init(&sem, 0, 1);
// flag 0 = same-process threads only

sem_wait(&sem);  // acquire
/* critical section */
sem_post(&sem);  // release
07 Java Synchronization
Built-in Monitor

Every Java object has an associated lock and a single unnamed condition variable. Declaring a method synchronized forces a thread to acquire the object's lock before executing it.

Entry Set & Wait Set
🔷Entry Set: Threads blocked trying to acquire the lock are queued here.
🔷Wait Set: A thread inside a synchronized method can call wait() to release the lock and move here until notified.
🔷notify() moves one arbitrary thread from the wait set back to the entry set to re-compete for the lock.
Tool Description When to Use
synchronized Built-in keyword; automatically acquires and releases the object's intrinsic lock. Simple cases; cleaner, less error-prone code.
ReentrantLock Explicit lock object from java.util.concurrent.locks. Supports fairness policies and reentrancy. Must be unlocked in a finally block. When you need fairness guarantees, timed lock attempts, or multiple conditions.
Semaphore Classic counting semaphore. acquire() and release(). acquire() throws InterruptedException, so use finally to guarantee release. When you need to allow N threads into a section simultaneously.
Condition Named condition variables created via lock.newCondition(). A ReentrantLock can have multiple, unlike the single wait set of synchronized. When threads must wait for different specific conditions on the same lock.
synchronized keyword 
public synchronized void insert(E item) {
  // lock acquired automatically
  /* critical section */
  // lock released automatically
}
ReentrantLock 
ReentrantLock key = new ReentrantLock();

key.lock();
try {
  /* critical section */
} finally {
  key.unlock(); // always runs
}
08 Beyond Locks: Future Approaches

As core counts increase, traditional locking becomes a bottleneck and is notoriously difficult to get right. New approaches seek higher levels of abstraction.

Transactional Memory

Borrows from database theory. The system executes an atomic { } block as a transaction — either all operations succeed, or all are rolled back. Deadlock is not possible because no locks are held explicitly.

OpenMP

A compiler-directive-based API. Developers mark a block as #pragma omp critical and the compiler handles the underlying locking automatically. Simplifies protecting shared data without manual mutex management.

Functional Programming

Languages like Erlang and Scala disallow mutable state. If variables are immutable, race conditions and the need for locks are eliminated by design — not by runtime control.

Enduring Principle

While the tools and abstractions evolve — from raw semaphores to monitors, from POSIX to Java, from locks to transactional memory — the fundamental goal remains the same: ensuring correct and efficient cooperation between concurrent processes in the presence of shared state.
