Inheritance & Polymorphism

Study notes compiled from lecture slides

01

Motivating Example — Employee System

Consider a payroll system that needs to handle different kinds of employees:

Employee ← shared: name, id, …
├── FullTimeEmployee ← fixed monthly salary
└── PartTimeEmployee ← paid by the hour

Without inheritance, you'd copy shared fields and methods into every class — causing duplication and maintenance headaches. Inheritance solves this.

02

Inheritance

  • Superclass (a.k.a. base class) — holds common data and operations shared by all subtypes
  • Subclass (a.k.a. derived class) — holds specific data and operations unique to that type
  • Subclass automatically inherits all accessible members of the superclass

Java

class Animal {
  void eat() {
    System.out.println(
      "This animal eats food."
    );
  }
}

class Dog extends Animal {
  void bark() {
    System.out.println(
      "The dog barks."
    );
  }
}

Dog d = new Dog();
d.eat();   // inherited ✅
d.bark();  // own method ✅

Python

class Animal:
  def eat(self):
    print(
      "This animal eats food."
    )

class Dog(Animal):  # inherits
  def bark(self):
    print(
      "The dog barks."
    )

d = Dog()
d.eat()   # inherited ✅
d.bark()  # own method ✅
03

Benefits of Inheritance

  • Code reuse — write shared logic once in the superclass, not in every subclass
  • Reduced redundancy → easier to maintain; fix a bug once, fixed everywhere
  • Better real-world modeling — class hierarchies often mirror domain concepts naturally
  • Foundation for polymorphism — enables writing flexible, general-purpose code (more below)
04

Multiple Inheritance & MRO (Python)

Python supports inheriting from multiple parent classes simultaneously. When a method name exists in more than one parent, Python uses the Method Resolution Order (MRO) — checking classes from left to right in the inheritance list.

class A:
  def greet(self): print("Hello from A")

class B:
  def greet(self): print("Hello from B")

class C(A, B):   # A is checked first (left→right)
  pass

C().greet()   # → "Hello from A"
Java does not support multiple class inheritance (to avoid the "diamond problem"), but a class can implement multiple interfaces.
05

Inheritance vs. Encapsulation

Inheritance

Subclass wants to reuse superclass code — needs visibility into its internals.

Encapsulation

Changes to one class's implementation should not break other code, including subclasses.

These two goals are in tension. In practice: find a pragmatic balance. Use access modifiers to expose only what subclasses truly need.

ModifierSame ClassSubclassSame PackageEverywhere
private✅❌❌❌
(default)✅❌*✅❌
protected✅✅✅❌
public✅✅✅✅
Python uses naming convention instead: name = public, _name = protected (by convention), __name = name-mangled (pseudo-private).
06

Method Overriding

A subclass re-implements a superclass method with the exact same signature to provide specialized behavior.

// Java
class Animal {
  void eat() { System.out.println("This animal eats food."); }
}

class Dog extends Animal {
  @Override
  void eat() { System.out.println("The dog eats treats."); }
}

Dog d = new Dog();
d.eat();
The dog eats treats.
Overriding

Same method name and same parameter types.

Involves a superclass–subclass relationship.

Resolved at runtime (dynamic binding).

Overloading

Same method name but different parameter types/count.

Can happen in the same class.

Resolved at compile time.

Why @Override? It tells the compiler to verify that you're actually overriding a superclass method — if you mistype the method name, it becomes an error instead of silently creating a new unrelated method.
07

Referencing the Superclass with super

Subclasses can still call the original superclass implementation using super.

Java

class Dog extends Animal {
  @Override
  void eat() {
    // call parent first
    super.eat();
    System.out.println(
      "...and then some treats."
    );
  }
}

Python

class Dog(Animal):
  def eat(self):
    # call parent first
    super().eat()
    print(
      "...and then some treats."
    )
08

Abstract Classes

Sometimes a superclass method cannot be meaningfully implemented at that level — it depends on the specific subclass. An abstract class lets you declare such methods without implementing them.

Employee ← abstract: calculatePay() has no implementation here
├── FullTimeEmployee ← returns fixed monthly salary
└── PartTimeEmployee ← returns hoursWorked × hourlyRate
// Java
abstract class Employee {
  String name;
  int id;
  // No body — subclasses MUST implement this
  abstract double calculatePay(YearMonth month);
}
  • Must be declared with the abstract keyword
  • Can contain a mix of concrete and abstract methods
  • Cannot be instantiated — new Employee() would be a compile error
  • Subclasses must implement all abstract methods (or be abstract themselves)
❓ Why not just create an Employee object directly?
It doesn't make sense — you can't calculate pay without knowing whether someone is full-time or part-time. The abstract class enforces that only concrete subclasses (which know how to calculate pay) can be instantiated.
09

Class → Abstract Class → Interface

These three constructs form a spectrum of how much implementation they provide:

Class
All methods concrete
Abstract Class
Some methods abstract
Interface
All methods abstract
// Java Interface
interface Vehicle {
  int MAX_SPEED = 120;          // implicitly public static final
  void start();                  // implicitly public abstract
  void stop();
  int  getSpeed();
}

class Truck implements Vehicle {
  public void start()    { /* … */ }
  public void stop()     { /* … */ }
  public int  getSpeed() { /* … */ }
}
  • A class can implement multiple interfaces (solving Java's no-multiple-inheritance restriction)
  • Interfaces can extend other interfaces (sub-interfaces)
  • Key principle: "Program to an interface, not an implementation"
// Good: variable typed as the interface
List list = new ArrayList();
// Swapping ArrayList → LinkedList requires changing only ONE line
// All other code that uses 'list' remains untouched ✅
10

Polymorphism & Dynamic Binding

Polymorphism means "many forms" — an object of a subclass can be treated as an object of the superclass, and the correct method is chosen at runtime.

  • A superclass reference variable can point to a subclass object
  • When an overridden method is called, the actual class of the object determines which version runs — this is called dynamic binding
// What does this print?
class A {
  void print() { System.out.println("This is A"); }
}
class B extends A {
  @Override
  void print() { System.out.println("This is B"); }
}

A a = new B();  // declared as A, but actual object is B
a.print();
This is B
Even though the variable is typed as A, the runtime sees that the actual object is a B, so it calls B's version of print(). This is dynamic binding.

Polymorphism lets you write highly general code:

// Works for FullTimeEmployee, PartTimeEmployee, or any future subclass
for (Employee e : employees)
  var pay = e.calculatePay(YearMonth.now());

// Works for CheckingAccount, SavingsAccount, etc.
double transfer(double amount, Account other) {
  return other.deposit(withdraw(amount));
}
