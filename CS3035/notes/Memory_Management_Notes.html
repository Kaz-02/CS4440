01

Why Different Storage Types?

Not all data has the same lifetime or size requirements. Consider this C example:

int x = 10;                          // global → static storage

int* foo(int a[], int size) {
    int* bar = malloc(size * sizeof(int)); // heap → dynamic storage
    for (int i = 0; i < size; i++)       // i, size → stack (automatic)
        bar[i] = a[i] > x ? 1 : 0;
    return bar;
}

int main() {
    int arr[] = {5, 15, 8, 20};           // arr → stack (automatic)
    int* result = foo(arr, 4);
    return 0;
}
❓ Is bar stored on the stack or the heap?
Both — in different ways. The bar pointer variable itself lives on the stack (it's a local variable). But the memory it points to — allocated by malloc() — lives on the heap. When foo() returns, the pointer is gone, but the heap memory persists until you call free().

03

Managing Dynamic (Heap) Storage

Dynamic memory must be both allocated and released. Forgetting to release it causes a memory leak — your program slowly consumes all available RAM.

Allocation

  • Explicit: malloc() in C
  • Implicit: new BankAccount("John", 100)

Release

  • Manual: programmer calls free()
  • Automatic: garbage collector

Simple case — C

int n = 10;
int *p = malloc(n * sizeof(int));  // allocate on heap

// … use p …

free(p);  // release — don't forget this!
p = NULL; // good practice: avoid dangling pointer
A memory leak occurs when heap memory is allocated but never freed. The program's memory usage grows over time and can eventually crash the system.

list = new LinkedList(10);

free(head); // ❌ Only frees node [10] — node [20] leaks!
Calling free(head) only releases the first node. Node [20] and any further nodes are now unreachable and leaked. You must traverse the entire list and free every node.

The Dispose Pattern

Classes that internally allocate resources should expose a cleanup method so callers can properly release everything — not just the top-level pointer.

  • Common names: dispose(), close(), release(), destroy()
  • Applies to more than memory — also file handles, DB connections, network sockets, etc.
  • Caller is responsible for invoking it at the right time

06

Semi-automatic Release — with / using

Python and C# guarantee that dispose() is called automatically when a block exits — even if an exception occurs.

Python — with statement

with open("student_records.csv", "r") as file:
    data = file.read()
    …
# file.close() is called automatically here, even on exception ✅

C# — using statement

using (var writer = new StreamWriter("students.txt"))
{
    writer.WriteLine("Student ID: 12345");
    …
} // writer.Dispose() called automatically here ✅
Both Python and C# have garbage collection for memory, so with/using is mainly used for non-memory resources like files, sockets, and DB connections.
07

RAII — Resource Acquisition Is Initialization

A C++ pattern: tie resource lifetime directly to object lifetime using constructors and destructors.

  • Constructor — acquire the resource (open file, allocate memory, etc.)
  • Destructor — release the resource automatically called when object goes out of scope
  • Destructor is called deterministically — the moment the object leaves scope, not some time later

C++ RAII example

class FileHandler {
public:
    FileHandler(const string& path) {
        file = fopen(path.c_str(), "r"); // acquire in constructor
    }
    ~FileHandler() {
        if (file) fclose(file);           // release in destructor
    }
private:
    FILE* file;
};

void readData() {
    FileHandler fh("data.txt");  // file opened
    // … use fh …
} // ← destructor called here automatically — file closed ✅
RAII is the dominant resource management idiom in modern C++. Smart pointers (unique_ptr, shared_ptr) are built on this exact principle.
08

Fully Automatic — Garbage Collection

The language runtime periodically identifies memory that is no longer reachable and reclaims it automatically. The programmer never calls free().

Reference Counting GC

Each object tracks how many references point to it.

When count drops to 0 → object is freed immediately.

✅ Predictable, no long pauses

❌ Assignment & parameter passing are slower

❌ Cannot handle circular references

Used by: Python (primary GC), Swift, Rust (Arc)

Tracing GC (Mark & Sweep)

Periodically traces all reachable objects from root set.

Objects not reached → garbage → reclaimed.

✅ Handles circular references

✅ No overhead per assignment

❌ Occasional "stop-the-world" pauses

Used by: Java, C#, Go, JavaScript

Reference Counting — Step by Step

Every object has a counter tracking how many variables reference it:

b = object()   # object 2 created — ref count = 1
a = b           # a now points to object 2

Circular reference problem: If object A holds a reference to object B, and B holds a reference back to A, neither ever reaches zero — they leak forever. Python solves this with a secondary cycle detector.

GC by Tracing — Mark & Sweep

The GC starts from a root set (active variables, stack frames, global refs) and traces all reachable objects. Anything not reached is garbage.

A and H are in the root set → everything reachable from them is kept → L, M, N, O are unreachable → reclaimed by GC

  • Mark phase: traverse from roots, mark all reachable objects
  • Sweep phase: scan all heap memory, free anything unmarked
  • Solves circular references — if L→M→N→O→L cycle has no root, all four are reclaimed
