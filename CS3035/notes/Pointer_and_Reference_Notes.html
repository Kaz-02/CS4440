01

Value Semantics

With value semantics, a variable stores the actual data. When you assign one variable to another or pass it to a function, an independent copy of the data is made. Changes to one variable do not affect the other.

Python example (immutable)

x = 10
y = x       # y gets a copy of 10
x += 1      # x becomes 11
print(x)    # 11
print(y)    # 10  ← unchanged

Memory model

x
10 → 11
← own slot
y
10
← own copy

x and y are independent. Changing x has no effect on y.

Examples of value semantics: Python immutable types (int, float, str, tuple) and Java primitive types (int, double, boolean, …).
02

Reference Semantics

With reference semantics, a variable stores a reference (memory address) pointing to the data. Assigning to another variable or passing to a function copies the reference, not the data — so both variables point to the same object.

Python example (mutable)

a = [10]
b = a        # b gets a copy of the reference
a[0] += 1   # modifies the shared object
print(a)     # [11]
print(b)     # [11] ← also changed!

a and b both point to the same list. Mutating through either name affects both.

Examples of reference semantics: Python mutable types (list, dict, set, most objects) and Java class types (all objects, arrays).

04

References in C: Pointers

All C types use value semantics by default. C has no built-in reference semantics — instead, when a reference is needed, you use a pointer: a variable that explicitly stores the memory address of another variable.

int x = 10; declare an integer variable x, initialize to 10
int *p; declare p as a pointer to int — stores an address, not an int
p = &x; & = "address of" — p now holds the memory address of x
int y = *p; * = "dereference" — follow the address to get the value (10); assign to y
*p = 20; dereference p and write 20 — this changes x's value to 20!
printf("p: %p, *p: %d\n", p, *p); // address, 20 %p prints the address stored in p
printf("x: %d, y: %d\n", x, y); // 20, 10 x was modified through *p; y still holds the original copy

Memory model after the code above:

20
← *p writes here
y addr: 0x104
10
← value copy made before *p=20
p addr: 0x108
0x100
← stores address of x
05

Pointer Operators

Operator
& — address of

Returns the memory address of a variable.
&x gives a pointer to x.

Operator
* — dereference

Follows a pointer and accesses the value at that address.
*p reads/writes the value that p points to.

Operator
-> — member via pointer

Access a struct field through a pointer. p->x is shorthand for (*p).x.

Struct pointer example 
typedef struct { int x; int y; } Point;

Point p1 = {1, 2};
Point *p  = &p1;

// Two equivalent ways to access members:
printf("%d, %d", p->x, p->y);       // → 1, 2
printf("%d, %d", (*p).x, (*p).y);  // → 1, 2 (same thing)
p->field is just syntactic sugar for (*p).field. You'll see -> everywhere in C code that works with structs through pointers.

Pointers & Arrays

Arrays and pointers are closely related in C, which is a common source of confusion:

FeatureArrayPointer
StoresA fixed block of memory (all elements)A single address
Memory allocatedSpace for all elementsSpace for one address (4 or 8 bytes)

Array "decay" to pointer 
int arr[] = {1, 2, 3};
int *p = arr;  // arr "decays" to &arr[0] — pointer to first element

// These are all equivalent:
arr[0]          // = 1
p[0]            // = 1
*p              // = 1
*(p + 0)       // = 1

⚠ Array decay is a common pitfall. When you pass an array to a function, it decays into a pointer — the function has no idea how long the array is. Always pass the size as a separate parameter.
07

When to Use Pointers

Use case 01
Call-by-reference

Allow a function to modify a variable in the caller's scope. Pass &x and use *x inside the function.

Use case 02
Memory efficiency

Passing a pointer to a large struct is cheap (8 bytes). Copying the struct directly can be very expensive.

Use case 03
Data structures

Linked lists, trees, graphs — each node holds a pointer to the next. Without pointers, these structures are impossible.

Use case 04
Dynamic memory

Allocate memory at runtime with malloc() — the only way to create data whose size isn't known at compile time.

Call-by-reference example 
// Without pointer: does NOT modify the original
void bad_increment(int x) { x++; }  // x is a copy

// With pointer: modifies the original
void increment(int *x) { (*x)++; }

int n = 5;
bad_increment(n);        // n still 5
increment(&n);           // n is now 6
08

Dynamic Memory Allocation

Static arrays have a fixed size set at compile time. Dynamic allocation with malloc() lets you request memory at runtime based on a value not known until the program runs.

malloc + free 
#include <stdlib.h>   // provides malloc() and free()

int n = 10;
int *p = malloc(n * sizeof(int));  // allocate 10 ints = 40 bytes
// malloc returns a pointer to the first byte, or NULL on failure

if (p == NULL) {
  fprintf(stderr, "allocation failed\n");
  return 1;
}

// use p as an array...
for (int i = 0; i < n; i++) p[i] = i * 10;

free(p);   // release memory back to the system — always do this!
p = NULL;  // good practice: null out pointer after freeing

Stack memory (automatic)

int arr[10];  // fixed at compile time
// freed automatically when function returns

Size must be a compile-time constant. Fast, automatic cleanup.

Heap memory (dynamic)

int *arr = malloc(n * sizeof(int));
// size can be any runtime value
free(arr);  // YOU must free it

Size can be determined at runtime. Must be freed manually.

09

Pointer Arithmetic

You can do arithmetic on pointers. When you write p + i, C automatically scales i by the size of the pointed-to type — so p + 1 moves to the next element, not the next byte.

int n = 5;
int *p = malloc(n * sizeof(int));

for (int i = 0; i < n; i++) {
  *(p + i) = i * 10;          // pointer arithmetic: *(p+0)=0, *(p+1)=10, …
}
// Equivalent to: p[i] = i * 10;

// Walk a pointer through an array:
int *cur = p;
while (cur < p + n) {
  printf("%d ", *cur);
  cur++;                          // moves 4 bytes forward (size of int)
}

Dangers of Pointers

Pointers are powerful but are the root cause of many of C's infamous bugs and security vulnerabilities. Every C programmer must know these hazards:

⚠ Wild Pointer

Using a pointer that was never initialized. Points to a random memory location — reads or writes will corrupt memory.

int *p;   // uninitialized!
*p = 5;  // ← undefined behavior
⚠ Dangling Pointer

Using a pointer after the memory it pointed to has been freed. The memory may have been reallocated for something else.

free(p);
*p = 5;  // ← use-after-free bug
⚠ Memory Leak

Forgetting to call free() after malloc(). Memory stays allocated for the life of the program, eventually exhausting RAM.

int *p = malloc(100);
// ... forgot free(p);
// ← memory leak
⚠ Buffer Overflow

Writing past the end of an allocated array. Overwrites adjacent memory — the source of countless real-world exploits (Heartbleed, Morris Worm, etc.).

int arr[3];
arr[5] = 99; // ← out of bounds!
⚠ Arithmetic Errors

Miscalculating pointer offsets leads to reading or writing the wrong memory location — often silently.

int *p = arr;
*(p + 100) = 0; // ← way out of range
✅ Best Practices

Always initialize pointers to NULL. Set to NULL after free(). Check malloc return. Use tools like Valgrind or AddressSanitizer to detect leaks.

int *p = NULL; // safe init
free(p);
p = NULL;     // avoid dangling
